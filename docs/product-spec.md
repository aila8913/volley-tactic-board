# 產品規格：比賽記錄與分析

> 這份文件描述「要做什麼、為什麼」，不寫程式碼細節。技術實作對應到 [db-schema-spec.md](./db-schema-spec.md)
> 和 [api-spec.md](./api-spec.md)。總覽見 [spec-index.md](./spec-index.md)。

## 背景

現有的戰術板（`/`）只處理賽前排兵布陣。這份規格擴充三個新階段：**賽中即時記錄**、**賽後影片補填**、
**防守數據分析**。三者共用同一套資料（同一場比賽的同一批球），只是「什麼時候記錄」不同。

## 使用情境

只有**一個人**負責記錄（教練或助理），不需要多人同時操作、不需要即時同步給其他裝置。比賽現場網路可能不穩，
記錄動作要先存在裝置本機，網路恢復後再同步到後端，避免漏記。

## 功能一：賽中即時記錄

比賽進行的同時，記錄每一分、每一球：

- 在球場圖上**點擊兩個座標**（球從哪裡來、往哪裡去），不是選區域編號——使用者明確要求要精確座標。
- 選擇是哪個球員、做了什麼動作（發球/接發/舉球/攻擊/攔網/防守）。
- 加標籤：有預設清單（例如 `#強扣`、`#失誤`）方便快速點選，也可以自己輸入新標籤。
- 一分結束時記錄比分。

## 功能二：賽後影片補填

跟賽中記錄是**同一個畫面**，不是分開的頁面。差別只在於：

- 比賽建立時可以貼上 YouTube 連結（`video_url`）。
- 有連結時，畫面上會出現 YouTube 播放器；點擊球場記錄球路時，系統自動抓當前播放秒數存成
  `video_timestamp`，方便回頭核對。
- 沒有連結時，這部分區域不顯示，記錄方式跟賽中模式完全一樣。
- 用途：賽中來不及記、或記錄有缺漏時，賽後看影片回填或修正。

## 功能三：防守數據分析

後端要能依下面的分類彙總統計（沿用排球教練常用的記錄習慣）：

```
一傳 (對方發球時的接球)
└── 接發: 到位率、到位球數、總接球數

防守 (對方攻擊時的接球)
├── 總到位率（以下三類合計)
├── 接扣(S)：對方強攻扣球
├── 嗆司：對方送來的輕鬆球／機會球（待確認確切定義）
└── 接吊(T)：對方吊球、輕拍球
    每一類都要算：到位率、到位球數、總接球數
```

「到位率」的定義：每一球記錄時要有一個**品質評分**（沿用排球記錄慣例的 0–3 分制：0 分 = 直接失誤,
3 分 = 完美到位讓舉球員能自由選擇戰術）。到位率 = 品質評分達到一定門檻的球數 / 總接球數。門檻值
（例如 2 分以上算到位）目前先抓常見慣例，未來如果教練的標準不同可以調整。

## 待確認事項

- **嗆司**的精確定義（對方失誤送過來的球？還是主動放的輕鬆球？）會影響怎麼把一球歸類成這一類，需要在
  正式串接分析功能前跟使用者確認。
- 到位率的品質門檻（幾分以上算到位）目前是假設值，非正式規則。
