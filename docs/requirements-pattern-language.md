# 需求分析：Alexander 式 Pattern Language

> 用 Christopher Alexander《The Timeless Way of Building》與後人的設計模式概念，
> 把這個 app 的**需求**（而非程式碼架構）拆成一套模式語言。
> 由 fable-advisor 於 2026-07-17 產出。這是**需求層**的思考文件，跟
> `product-vision.md`（定位）、`recording-cost-budget.md`（記錄成本）、
> `event-grammar-spec.md`（統計可推導性）互為表裡。

## 幾個術語（Alexander 的用法）

- **Pattern（模式）**：某個脈絡下反覆出現的問題，加上一個能消解該問題**核心衝突**的解法。
  重點不是解法本身，而是它消解了什麼張力。
- **Forces（力）**：該脈絡下互相拉扯的真實壓力——時間、注意力、成本、期待。
  **沒有互相矛盾的 forces，就沒有真正的問題，只有待辦事項。**這是整份分析的靈魂。
- **Pattern language（模式語言）**：patterns 不是清單而是網絡——每個 pattern 的解法落地後
  產生的新脈絡（resulting context），正是下一層 pattern 的 context。
- **Quality without a name / wholeness（無以名之的品質／整體性）**：當所有 forces 被真正
  **消解**（resolved）而不是**壓抑**（suppressed，硬吞下去假裝不存在）時，結構會有一種活的感覺。

## Pattern Language 總覽

四層，箭頭表示「上層的 resulting context 開啟了下層」：

```
第 0 層（產品定位）
  P1 平板取代紙本記分表
      │
      ├─────────────┬──────────────────┬──────────────┐
      ▼             ▼                  ▼              ▼
第 1 層（結構性拆解）
  P2 懸崖式雙層記錄   P4 名單是歷史事實、  P5 三板分離、   P7 離線優先的
  （簡易/進階硬分界）    身分是跨場錨點      快照解耦        單人 PWA
      │                   │               │
      ▼                   │               │
第 2 層（資料語意）        │               │
  P3 種子＋序列           │               │
  （能推導的就不存）       │               │
      │                   │               │
      └───────┬───────────┘               │
              ▼                           │
第 3 層（價值兌現）                        │
  P6 粗粒度球線分布（wow 點）◄─────────────┘
      │
      ▼
  #76 比賽生命週期閉環（排陣→記錄→分析→回頭修正排陣，規劃中）
```

網絡性的關鍵連結：

- **P1 → P2**：「操作成本不得高於紙筆」撞上「數據要細才有分析價值」，逼出雙層分界。
- **P2 → P3**：簡易版只記一筆決定球 → 統計還算得出什麼？逼出「統計＝events 的純函數」。
- **P3 + P4 → P6**：座標語意＋身分模型定了，球線分布與跨隊分析才有原料。
- **P1 → P7**：「場邊、單人、網路不穩」的情境直接逼出離線契約與 PWA。

## P1：平板取代紙本記分表（根 pattern）

- **Context**：業餘/系隊排球圈。球經是被指派拿紙本記分表的人（不是數據愛好者）；隊上另有
  一位認真夥伴曾用 Excel＋慢速回放手算數據。市面工具（Data Volley 等）以職業攻擊效率為核心、
  昂貴、英文介面。
- **Forces**：數據有價值 ⟷ 對球經而言記錄是苦差，任何比紙筆麻煩的工具都會被放棄；
  想做防守差異化 ⟷ 使用者期待基本功能一樣不缺（不能是 defense-only 殘缺品）。
- **Problem**：怎麼讓一個不是為了數據而來的人，自然產出可分析的數據？
- **Solution**：楔子＝「取代紙本記分表」——切入球經**本來就得做**的記錄，數據分析是記錄後
  自動產出的獎勵；差異化＝「**新增**」防守維度（基本記錄照常，只是把防守拉成一級公民）。
- **Resulting Context**：「成本不得高於紙筆」的承諾開啟 P2；「場邊單人平板」開啟 P7；
  「一帳號記多隊、同人跨隊」開啟 P4。
- **落點**：`docs/product-vision.md`（定案）。

## P2：懸崖式雙層記錄（簡易版／進階版硬分界）

- **Context**：P1 落地後，一個人在場邊即時記錄。一個 rally 有 6–10 次觸球、只有幾十秒。
- **Forces**：記錄要**細**（座標、每觸品質分、子分類） ⟷ 記錄要**快**（live 期間視線離場預算≈0）；
  想開中間層漸進取捨 ⟷ 節拍純度（多記一筆就破壞「一分＝一節拍」的韻律）。
- **Problem**：單人賽中的時間預算，物理上塞不下產品承諾的記錄粒度。
- **Solution**：`docs/recording-cost-budget.md` 的核心洞察——這條線是**懸崖不是斜坡**。
  簡易版：死球空檔記恰好一筆決定球（UX 判準＝俐落簡約）；進階版：賽後影片補填完整觸球鏈
  （UX 判準＝節奏遊戲式體驗）。兩層寫**同一張 events 表**（欄位全 nullable、`source: live|review`）。
  PO 拍板不開 simple+（節拍純度 > 資料豐富度）。
- **Resulting Context**：(1)「沒看到」escape valve——整筆降級成 outcome-only rally，待補清單可推導；
  (2) 記太少 → 開啟 P3；(3) 進階版「節奏遊戲」承諾目前只是判準、還沒介面兌現（#51/#21，M4）。
- **落點**：簡易版已實作（`useScoreSheet.ts`、`ScoreSheetCourt`/`RadialMenu`）；進階版 UI 尚缺。

## P3：種子＋序列（能推導的就不存）

- **Context**：簡易版只留稀疏資料（每分一 winner、至多一筆決定球）。統計頁要算比分趨勢、
  side-out%、輪次得失、到位率。
- **Forces**：直接存統計數字最省事 ⟷ 資料補不回來（門檻一改、原始評分沒存就永遠改不了）；
  欄位越多越保險 ⟷ 每個冗餘欄位都是可能失同步的第二真相。
- **Problem**：哪些必須存、哪些必須不存，才能讓統計永遠可重算、可換門檻、可拆維度？
- **Solution**：「每個統計都必須是 events 的純函數」（`docs/event-grammar-spec.md`）。比分/發球方/
  輪轉由種子 `sets.firstServer`＋`rallies.winner` 序列**重播**。刻意例外：`rallies` 存「該分開始前」
  比分快照，讓每筆 rally 能獨立還原情境。
- **Resulting Context**：統計層可誠實「等原料」（分析頁用誠實空狀態，不假資料）；開啟 P6。
  B4 輪次統計缺口催生 `lineups` 表——pattern 網絡自我修補的例子。
- **落點**：已實作（schema sets/rallies/events + `event-grammar-spec.md` 對照表）。

## P4：名單是歷史事實、身分是跨場錨點

- **Context**：單帳號 ≠ 單隊；同一人跨隊出現，「同一個你在不同隊數據會變」是一級分析情境。
- **Forces**：想跨場追同一人 ⟷ 每場名單是歷史事實（那場穿幾號、打什麼位置，可能不同）；
  身分要能刪能改 ⟷ 刪身分不能連帶抹掉歷史比賽紀錄。
- **Problem**：一個「人」同時要當歷史快照、跨場錨點、缺席者（對手），單張表裝不下。
- **Solution**：兩層——`players`＝一場的一列名單（不可變）；`people`＝跨場身分錨點
  （`players.personId` nullable FK、`onDelete: set null`，刪身分只斷連結不抹歷史）；
  `teams`＝分組標籤。對手用 `events.side=away, playerId=null` 表達，不造假身分。
- **Resulting Context**：開啟「跨隊同人分析」視圖（規劃中）；帶出去重 UX、身分錯連修復等新 force，
  尚無 pattern 接住。
- **落點**：schema 已 live（players/people/teams）；應用層尚缺（openapi/前端讀寫、去重 UX、回填）。

## P5：三板分離、快照解耦（輪轉表／戰術板／計分表）

- **Context**：三個功能面早期共用一坨全域 Zustand store。
- **Forces**：資料共享方便 ⟷ 全域 store 跨比賽汙染（#115/#117/#118/#119 家族之源）；
  live 連動直覺 ⟷ 戰術板本質是自由白板，硬同步會錯誤施加輪轉規則（歷史 libero 同步 bug）。
- **Problem**：三個功能對「球員站位」語意根本不同（規則狀態 vs 自由畫布 vs 逐局快照），
  共用 live 資料必然互相汙染。
- **Solution**：三個獨立 store（per-match keying），跨界用**顯式單向快照**（`enterTacticsLayout`
  一次性複製、之後解耦不回寫）；計分表擁有自己逐局的先發快照。
- **Resulting Context**：解掉汙染，但產生新力——「每次進戰術布置就覆蓋重拍」vs「想接續上次」，
  目前靠第二顆「編輯」按鈕繞過（**力被安排而非消解**，見評語）。開啟 #120 唯讀站位視圖。
- **落點**：大部分已實作（三個 hooks store、`TacticsBoard.tsx`）；#119/#120 收尾進行中。

## P6：粗粒度球線分布（wow 點）

- **Context**：產品需要一個被記住的畫面。教練賽前想知道「對手的球往哪走」來佈防。
  原料來自 P2 進階版座標＋P3 座標語意。
- **Forces**：熱區圖要有衝擊力 ⟷ 業餘影片單機位、座標估計必然不準；存區域編號省事 ⟷
  一旦只存區域，未來換切法就回不去；座標是最貴的記錄動作 ⟷ wow 點不能＝記錄地獄。
- **Problem**：資料必然粗糙、記錄必然昂貴的前提下，怎麼交付有戰術價值的球線圖？
- **Solution**：存精確座標、區域為衍生顯示值（聚合後個別誤差被洗掉，粗粒度夠佈防）；座標全歸
  進階版（賽後補填，賽中不依賴影片）；顯示層永遠可重新切分。
- **Resulting Context**：兌現後才開啟 **#76 生命週期閉環**（球線圖的價值不在「看」而在「跳回
  戰術板改佔位」）——這是 pattern language 最重要的未接線。
- **落點**：規劃中／尚缺。schema 已 advanced-ready（`events.fromX/Y/toX/Y` nullable）；
  座標互動（#21）、熱區渲染（#65）未做，排 M4。外部參考見 `docs/external-openvolley-reference.md`。

## P7：離線優先的單人 PWA

- **Context**：場邊平板記錄，體育館網路可能不穩；單人操作，不需多裝置同步；開發者是單人。
- **Forces**：記錄一筆都不能漏（漏了就是 P2 的永久降級） ⟷ 網路隨時會斷；
  原生 App 體驗好 ⟷ 第二套 codebase 的維護成本；給 TA 試用要零摩擦 ⟷ 離線通常要安裝。
- **Problem**：怎麼在不養第二套 codebase 下，兌現「先存本機、網路恢復再同步」的可靠性承諾？
- **Solution**：PWA 優先（加主畫面＋service worker＋IndexedDB 本機寫入佇列）；「離線儲存」與
  「記錄不遺失」是同一份工程（#75）；原生後路刻意敞開（React pin 在 Expo 相容的 19.1.0）。
- **Resulting Context**：離線佇列會開啟同步衝突、佇列重放順序（rally/event 有 FK 依賴鏈）、
  本機 uuid 鑄造——後者已提前落地（#117/#118 換 client-mintable uuid PK），為離線先鋪路。
- **落點**：**規劃中／尚缺，目前最大的承諾落差**。`docs/product-spec.md` 承諾「先存本機、網路恢復
  後同步」，但現況前端直接打 API，#64 記錄「背景寫入失敗不 reconcile」——網路一斷比 localStorage
  時代更易丟資料。#75 擋在部署（#26）前面。

## Alexander 式整體性評語

**活的地方（forces 被真正消解）**

1. **P2 的懸崖洞察是全語言的拱心石**——把看似 UI 調校的問題重新定義成物理極限問題，兩股力
   （快 vs 細）各自被給了完整的家，而非折衷成兩邊難受的中間態。
2. **P3「能推導的就不存」是活的骨架**——新需求一直能從既有結構推導而不用加欄位（「沒看到」
   待補清單、發球方情境過濾零成本），是結構活著的證據。
3. **P4 兩層身分模型**乾淨分層、`set null` 尊重歷史，additive 可逆。

**勉強的地方（forces 被壓抑或懸置）**

1. **P7 是目前最大的「假牆」**：spec 承諾離線可靠，實作卻是「網路好時一切正常」——力被暫時
   忽略（#64/#75 誠實記錄了，但記錄≠消解）。**M1 之後應優先補這塊，別拖到部署之後。**（#75/#64/#26）
2. **P6（wow 點）是語言中心卻還是個洞**——防守差異化、座標語意、進階記錄最終都指向球線分布
   與 #76 閉環，兩者都在 M4。判斷新需求的試金石：**問「它離 P6/#76 的兌現更近還是更遠」。**
3. **P5 的快照有一個被安排而非消解的力**：「覆蓋重拍」vs「接續上次」靠兩顆語意不同的按鈕
   並存，使用者得知道差別才不會誤按丟工作。#134 討論時值得問：兩個入口能否收斂成一個不需
   使用者先理解的動線。
4. **進階版「節奏遊戲」目前只是形容詞**：#51/#21 的介面還沒原型。建議用
   `docs/recording-cost-budget.md` 的原語成本表（定向/定位/精準 tap）反推進階版操作預算，
   讓「節奏遊戲」從口號變成可量測判準——就像簡易版當初做過的那樣。

整體而言：這個專案的需求結構**罕見地誠實**——每個大張力都有文件正面命名它，空缺處用「誠實
空狀態」而非假資料遮掩。milestone 順序（M1 去汙染收尾 → 離線 → M4 進階版）基本符合「每一步都
修補整體」的方向，唯一要盯住的是別讓 P7 的承諾落差拖到部署之後才還。
