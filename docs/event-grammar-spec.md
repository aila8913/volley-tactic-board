# 事件文法領域模型（統計反推對照表）

> **這份文件回答一個問題**：產品承諾過的每一個統計數字，能不能從我們現在存的資料
> 推導出來？推不出來的，就是 schema 的表達力缺口。
>
> 對應 issue [#73](https://github.com/aila8913/volley-tatic-board/issues/73)（`area:product`，
> 五個產品主題裡最優先、最不可逆的一個）。上位脈絡見 [`product-vision.md`](./product-vision.md)、
> 功能規格見 [`product-spec.md`](./product-spec.md)、執行步驟見 [`product-deep-dive-plan.md`](./product-deep-dive-plan.md) 的 T1。
> 定案於 2026-07-09 的產品討論。

## 為什麼要先做這件事

**核心原則：每個統計都必須是 events 的純函數。**

「純函數」在這裡的意思是：一個統計數字（例如「3 號的接發到位率」）應該完全由「我們
記錄下來的原始事件」算出來，不需要另外存一份。這樣做的理由是**資料補不回來**——

- 如果我們把「到位率」直接存成一個數字，之後想改到位門檻（例如從「2 分以上算到位」
  改成「1 分以上」），就只能重算，而且**過去的比賽再也改不了**，因為原始的每球評分
  沒存下來。
- 反過來，只要每球的原始評分（`quality`）有存，到位率就永遠能重算、能換門檻、能拆維度。

所以 schema 設計的判準不是「畫面上要顯示什麼數字」，而是**「要支撐這些數字，最少要
記下哪些原始事件欄位」**。這份文件就是把這件事做到底：窮舉統計 → 逐條寫推導式 →
標出推不出來的缺口。

**一個關鍵發現先講**：現行 `events` 表其實已經相當有表達力——它早就有 `ballType`
（接發/接扣/嗆司/接吊）、`quality`（0–3 到位分）、四個座標欄、`tags[]`。所以下面很多
缺口**不是「schema 少了欄位」，而是「簡易版記錄流程還沒去填那些欄位」**。這個區別很
重要，因為它把問題從「要不要改資料庫」轉成「賽中來不來得及記」——後者正是 T2（#74
記錄成本預算）要回答的。

## 現行資料模型（推導的原料）

巢狀關係：`Match（比賽）→ Set（局）→ Rally（一分）→ Event（一球）`，另有 `Player（名單）`。
以下只列跟統計推導有關的欄位（完整定義見 `lib/db/src/schema/`）。

| 表          | 統計會用到的欄位                                                                                                                                                                                                                              |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **sets**    | `firstServer`（home/away，唯一必存的種子）                                                                                                                                                                                                    |
| **rallies** | `rallyNumber`、`homeScore`/`awayScore`（該分開始前比分）、`winner`（home/away）                                                                                                                                                               |
| **events**  | `side`（home/away）、`playerId`（nullable）、`action`（serve/receive/set/attack/block/dig）、`ballType`（nullable：serve/spike/tip/chance）、`quality`（nullable：0–3）、`fromX/fromY/toX/toY`（nullable）、`tags[]`、`source`（live/review） |
| **players** | `number`、`role`（S/OH/MB/OPP/L）                                                                                                                                                                                                             |

**架構原則（已定，理解推導式的前提）**：所有「可以算出來」的量都不存，只存**種子＋
序列**再重算。具體說，比分、發球方、輪轉這些都不是欄位——它們是「`firstServer` 這個
種子 + 每個 rally 的 `winner` 序列」重播出來的。唯一推不出來、所以必須存的種子就是
`sets.firstServer`（誰先發）。這個「能推導的就不存」原則，本身就是這份文件的縮影。

## 統計反推對照表

三欄：**統計 ⇐ 推導式 ⇐ 所需欄位 / 缺口**。「可推」= 現有資料算得出來；「缺口」= 欄位在
但沒填、或欄位根本不存在。門檻預設沿用 spec：**到位 = `quality >= 2`**（暫定，可調）。

### 群組 A — 比分與局面（現有資料即可，全部可推）

| 統計                   | 推導式                                                                              | 所需欄位 / 狀態                       |
| ---------------------- | ----------------------------------------------------------------------------------- | ------------------------------------- |
| A1 每局比分            | 對每個 set，`我方分 = count(rally where winner=home)`、`對方分 = count(where away)` | `rallies.winner` ✅ 可推              |
| A2 局數比（大比分）    | 對每 set 用 A1 判誰高，計數                                                         | `rallies.winner` ✅ 可推              |
| A3 比分趨勢 / 士氣曲線 | 依 `rallyNumber` 排序畫 `(homeScore, awayScore)` 折線                               | `rallies.homeScore/awayScore` ✅ 可推 |
| A4 連續得分（runs）    | `winner` 序列的最長連續同值段                                                       | `rallies.winner` ✅ 可推              |

### 群組 B — 發球權與 side-out（種子＋序列重播）

| 統計                            | 推導式                                                          | 所需欄位 / 狀態                                                                            |
| ------------------------------- | --------------------------------------------------------------- | ------------------------------------------------------------------------------------------ |
| B1 每球發球方                   | 以 `firstServer` 為種子重播：接發方贏球（side-out）時發球權易手 | `sets.firstServer` + `rallies.winner` ✅ 可推                                              |
| B2 Side-out %（我方接發成功率） | `count(我方接發 且 我方贏) / count(我方接發)`                   | 由 B1 推出「該球我方是否接發」+ `winner` ✅ 可推                                           |
| B3 我方發球得分率（破發）       | `count(我方發球 且 我方贏) / count(我方發球)`                   | 同上 ✅ 可推                                                                               |
| B4 各輪次（1–6）得失分          | 重播輪轉（side-out 贏才輪轉一位）得到每球我方輪次 index，再分組 | `firstServer`+`winner` 可推**輪次編號**；但對不到「實際哪六人站哪」→ 見缺口 G2 ⚠️ 部分缺口 |

### 群組 C — 動作分布與球員統計（現有簡易版資料）

| 統計                                   | 推導式                                                                         | 所需欄位 / 狀態                                       |
| -------------------------------------- | ------------------------------------------------------------------------------ | ----------------------------------------------------- |
| C1 動作分布                            | `count(events) group by action`                                                | `events.action` ✅ 可推                               |
| C2 每人每動作得/失分矩陣（＝現行畫面） | group by `playerId, action`；得 = 該 event 所屬 rally `winner`=我方，失 = 對方 | `events(playerId, action)` + `rallies.winner` ✅ 可推 |
| C3 評價分布（0/1/2/3 各幾球）          | `count(events) group by quality`                                               | `events.quality` — 欄位在，**簡易版沒填** ❌ 缺口     |

> **C2 的語意侷限（重要）**：簡易版一個 rally 只記一個「決定球」event，所以這張矩陣是
> **「決定球歸屬」統計**，不是「每次觸球」統計。同一球的接—舉—攻三個人裡，只有最後決定
> 勝負那個被記到。這不是 bug，是簡易版的刻意取捨；進階版（多 event/rally）才有完整觸球鏈。

### 群組 D — 防守數據分析（功能三，產品差異化）★核心缺口

到位率類統計，每一類都要算三個數：**到位率、到位球數（分子）、總接球數（分母）**。

| 統計                 | 推導式                                                                                         | 所需欄位 / 狀態                                     |
| -------------------- | ---------------------------------------------------------------------------------------------- | --------------------------------------------------- |
| D1 接發到位率        | `count(action=receive ∧ ballType=serve ∧ quality>=2) / count(action=receive ∧ ballType=serve)` | `events(action, ballType, quality)`                 |
| D2 防守總到位率      | 同式，`action=dig`（含所有 ballType 合計）                                                     | 同上                                                |
| D3 接扣(S)到位率     | `action=dig ∧ ballType=spike`                                                                  | 同上                                                |
| D4 嗆司到位率        | `action=dig ∧ ballType=chance`（嗆司定義待確認，見 spec）                                      | 同上                                                |
| D5 接吊(T)到位率     | `action=dig ∧ ballType=tip`                                                                    | 同上                                                |
| D6 舉球到位率        | `count(action=set ∧ quality>=門檻) / count(action=set)`                                        | `events(action=set, quality)`；**待評估**（見下方） |
| D7 掩護(cover)到位率 | `action=dig ∧ ballType=cover`（攔網後的防守）                                                  | 同上；`ballType` 需 additive 加新值 `cover`         |

> **D 群是整個產品的差異化——已定案歸進階版（賽後影片補填），不進簡易版。**
> `ballType`/`quality` 欄位都已在 `events` 表，缺的不是 schema，是「賽中來不來得及點」。
> 使用者已拍板：到位率（quality）與座標都只在**進階版**出現，簡易版維持「誰、做什麼動作、
> 得失分」不變。到位率**要分「舉球(set)」與「防守(receive/dig)」兩類**（D6 為舉球）。
> 防守子分類在原本三類（接扣 spike / 嗆司 chance / 接吊 tip）之外**新增第四類「掩護 cover」
> ＝攔網後的防守**（D7），對應 `ballType` 加一個 additive 新值 `cover`。
>
> **記錄方式（UX，非 schema）**：防守 quality 傾向「**用落點到舉球位的距離自動估一個預設值，
> 可手動覆寫**」來減輕負擔；但 2D 畫面呈現不了高度/弧線，距離估必然不準，所以手動覆寫是必要的。
> 這也印證 `quality` 是**「種子」不是「推導量」**（含距離以外系統算不出的因素，最終值必須存）。
> 自動估只是預設值，不改 schema——`quality` 仍是可手動設定的儲存欄位。可行性留 T2/T4 深入。
> **舉球(set) 的 quality 更複雜**（好壞取決於戰術意圖，快攻與高球的理想落點不同），標為
> **待評估的開放問題**，不擋 T1。

### 群組 E — 攻擊

| 統計                                                      | 推導式                                                        | 所需欄位 / 狀態                                                                                                                             |
| --------------------------------------------------------- | ------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| E1 攻擊次數 / 得分（粗）                                  | `count(action=attack)`、`count(action=attack ∧ 我方贏)`       | `events.action` + `rallies.winner` ✅ 可推（粗粒度）                                                                                        |
| E2 攻擊效率 `(kills − errors) / attempts`（排球標準指標） | 需每次攻擊的 outcome（得分 kill／失誤 error／被防起 in_play） | 基礎版只記終結球（得分/失分），「打進被對方防起」的 in_play 攻擊根本不會被記，所以 attempts 分母不完整 → 需進階版逐球記錄 ❌ 缺口（屬 #51） |

### 群組 F — 球線分布圖（wow 點）★缺口（座標）

| 統計            | 推導式                                                            | 所需欄位 / 狀態                                                       |
| --------------- | ----------------------------------------------------------------- | --------------------------------------------------------------------- |
| F1 球線分布熱區 | 聚合 events 的 `(fromX,fromY)→(toX,toY)`，顯示層再收斂成號位/熱區 | `events` 四個座標欄——欄位在，**簡易版不記座標**，屬進階版 #21 ❌ 缺口 |

> 定位文件已定調「**存精確座標、區域為衍生顯示值**」：座標是輸入，號位/熱區是顯示層算出來
> 的，不是另一個輸入欄位。這也是 #21「落點要精確座標還是號位」那題的答案——選精確座標。

### 群組 G — 換人 / 暫停（非擊球事件）→ 需 schema 結構決策

| 統計                       | 推導式                      | 所需欄位 / 狀態                                                                 |
| -------------------------- | --------------------------- | ------------------------------------------------------------------------------- |
| G1 換人次數（本局/全場）   | `count(換人事件)`           | ✅ **已補**：`substitutions` 表（#42/#97/#98），見文末〈落地進度對帳〉          |
| G2 任一 rally 當下在場陣容 | 起始先發陣容 + 換人序列重播 | ✅ **已補**：`lineups` + `substitutions` 均已建，可重播，見文末〈落地進度對帳〉 |
| G3 暫停與士氣關聯          | 暫停時點 vs A4 連續得分     | 暫停未記錄（#44）❌ 缺口                                                        |

> G 群是唯一涉及**新資料結構**的部分（不是「填已有欄位」而是「這些事件該放哪張表」）。
> 這一節的結構決策——換人/暫停進 `events` 還是專用表、以及 G2 需不需要新存「起始先發」——

## Schema 缺口與決策清單

結構複審（fable-advisor）的總推薦。**判準來自 #73 第一原則「資料補不回來」**：一個決定
若「選錯了難退」或「沒存的資料事後補不回來」，就必須現在拍板；反之能之後 additive 補的，
隨功能節奏做即可。

| 懸案             | 推薦落點                                                            | 不可逆性                                    |
| ---------------- | ------------------------------------------------------------------- | ------------------------------------------- |
| (a) 換人 #42     | **獨立表 `substitutions`**，時機存「當下比分快照」                  | 表可後補；**時機表示法必須現在定**          |
| (a) 暫停 #44     | **獨立表 `timeouts`**（形狀現在定、實作可晚）                       | 可後補 additive                             |
| (b) 發球子類 #51 | **events 加 nullable `serveType` enum**                             | 欄位 additive；enum 值清單要先鎖            |
| (b) 犯規類別 #51 | **先用 `tags[]`**（UI 給預設清單餵，別手打）                        | 可後補；tag 拼寫漂移會讓回填有損            |
| (b) Outcome #51  | **events 加 nullable `outcome` enum**（統計核心缺口，優先於子分類） | 欄位 additive；enum 值清單要先鎖            |
| (c) 起始先發     | **新表 `lineups`**，一局一 row、`setId` unique、六個 zone 欄位      | schema additive，但**沒存到的比賽永久缺角** |

### 為什麼非擊球事件不進 events（(a) 的核心理由）

`events` 有一條不變量：**一 row = rally 內的一次觸球**（`rallyId` notNull、`side`+`playerId`
指誰執行這球）。換人/暫停破壞它的每一條：

1. **它們發生在死球期間、不屬於任何 rally。** 更關鍵是寫入順序：`rallies.winner` notNull，
   所以 rally row 要等這分打完才 insert，但換人發生在「下一 rally 開始前」，那時下一個
   rally id 還不存在。硬塞就得把 `rallyId` 改 nullable——而 **notNull → nullable 是單向門**，
   放了 null 進去就再也收不緊。
2. **換人要記兩個球員**（in + out），events 只有一個 `playerId`→ 多型表膨脹。
3. **查詢稅永久**：混入非擊球 row 後，每條「events 純函數」統計都要多一個
   `WHERE action IN (擊球動作)` 過濾；且 **PG enum 加值易、刪值幾乎不可能**，加
   `"substitution"` 進 `event_action` 實質不可逆。

也不共用一張 `set_annotations`：換人/暫停唯一共通只有 `(setId, 時機)`，共用會逼出一堆
「有效性看 type 欄位」的空欄位，DB 層擋不了錯。兩張各約五欄的小表更乾淨。**除非**未來走向
「完整正式記錄表數位化」（牌/挑戰/傷停都記），再泛化成共用表——那是 additive。

### 時機為何存「比分快照」而非 rallyId（(a) 唯一的不可逆點）

存事件當下的 `(homeScore, awayScore)`，鏡射 `rallies` 表「存這分開始前比分」的設計。一局內
比分對唯一且嚴格遞增（每分只有一邊 +1），所以 `(3,5)` 精準對應「第 9 分開球前」，**單筆即可
獨立寫入佇列**、不依賴任何其他 row 先存在——契合 local-first 單筆 atomic insert。還原任一
rally 當下陣容 = 先發陣容 + 所有 `subHome+subAway <= rallyNumber-1` 的換人依序套用。

### 建議表形狀（描述，非最終 DDL）

- **`substitutions`**：`id`、`setId`（FK cascade）、`homeScore`、`awayScore`、`playerInId`、
  `playerOutId`（FK，`onDelete: set null`，同 events.playerId 理由）、`kind` enum（`regular` /
  `libero`；自由球員上/下各記一筆，in/out 對調）。
- **`timeouts`**：`id`、`setId`（FK cascade）、`homeScore`、`awayScore`、`side`（home/away）。
  形狀現在定案，實作等 #44 UI 動工再做。
- **`lineups`**：`id`、`setId`（**unique**，一局一 row）、`zone1PlayerId`…`zone6PlayerId`（六欄
  notNull，DB 層保證陣容必然完整）。只記我方（對手無名單，`firstServer` 已足夠推導）；自由
  球員不進先發（前端邏輯本就「L 從場邊出發」）。「六人不得重複」的約束 DB 表達不了，放
  Zod/應用層檢查。

### (b) #51 判準：可聚合的走 typed 欄位，開放清單的走 tags

一句話：**會出現在統計 GROUP BY / 分母分子裡的值 → nullable typed 欄位（封閉清單用 enum）；
描述性、長尾、使用者自訂的 → `tags[]`。** 這跟 `ballType` 從 tag 升格成 enum 是同一個先例。
`outcome`（這球如何終結這分）是最重要的一項——`rallies.winner` 只知道誰贏、不知道哪次觸球
怎麼終結，攻擊決定率/發球直接得分率/失誤率全靠它。`serveType`/`outcome` 的 **enum 值清單要在
本文件統計對照表定案時一起鎖**（加值 additive、刪值不行，寧可先窄後寬）。

### 分界總結

**現在必須拍板（不可逆 / 資料一去不回）：**

1. 非擊球事件**不進** `event_action` enum（PG enum 刪值不可能）。
2. 換人/暫停時機 = **當下比分快照**（沒存就永遠補不回「第幾分換的人」）。
3. **`lineups` 要在開始記「真的在乎的比賽」之前上線**（沒存先發，該場輪次統計永久缺角）。
4. `serveType` / `outcome` 的 **enum 值清單**在統計對照表定案時一起鎖。

**之後 additive 補（低風險，隨功能節奏）：** `timeouts` 整張表、`substitutions.kind` 加值、
犯規類別從 tags 升格成欄位（前提：UI 用預設清單餵 tags）、events 加更多 nullable 統計欄位、
未來多種 set 層標記再把 timeouts 泛化成共用註記表。

## 使用者決策（PO 拍板）

T1 推導與結構複審攤出的產品價值取捨。以下記錄使用者已定案與仍待決定的部分。

### 已定案（2026-07-09）

1. **`lineups`（起始先發）→ 早補。** 這是唯一「不補就永久缺角」的項目，優先納入。代價是
   開局多一個「排先發六人」的記錄步驟（成本細節交 T2 #74）。
2. **到位率（`quality`）與座標 → 只在進階版（賽後影片補填）出現，簡易版不做。** 簡易版維持
   現狀（點人→選動作→得失分）。這同時把 D 群（防守差異化）與 F 群（球線分布 wow 點）都
   定位成進階版功能。
3. **到位率要分「舉球(set)」與「防守(receive/dig)」兩類**（對照表新增 D6 舉球到位率）。
   防守子分類再**新增第四類「掩護 cover」＝攔網後的防守**（D7），`ballType` additive 加新值 `cover`。
4. **防守 quality 的記錄方式（UX 方向）**：預設用「落點到舉球位的距離」自動估、可手動覆寫，
   以減輕記錄負擔；已知 2D 呈現不了高度/弧線，故手動覆寫必要。此為記錄介面設計、**不改
   schema**（`quality` 仍是可手動設定的儲存欄位），可行性留 T2/T4 深入。
5. **舉球(set) 的 quality → 標為待評估開放問題**（好壞取決於戰術意圖，比防守複雜），不擋 T1。
6. **`serveType`（發球種類）→ 純進階版欄位**，基礎版不分；進階版分 `跳發 / 飄球 / 跳飄`（enum
   鎖定，之後不夠再 additive 加值）。
7. **`outcome`（這球結果）→ 基礎版也存，不留 null。** 雖然基礎版的得分/失分能由
   `rallies.winner` + `events.side` 推導，但因為基礎版與進階版**共用同一張 events 表、基礎版資料
   會被進階版接續補填**，留 null 會讓「球續」與「終結球尚未填」兩種情況撞在一起。所以基礎版
   每筆事件都存 outcome（本就是記錄手勢第三步「選得失分」，寫入成本為零、不打斷節奏）。
   `outcome` enum 用**三個明確值**：`得分(point)` / `失分(loss)` / `球續(in_play)`，`null` 保留給
   「尚未填/未知」。**不變量**：完整記錄的 rally 其事件 outcome 皆非 null，恰好一筆得分或失分
   （終結球），其餘皆球續；基礎版一 rally 一筆，故只會是得分/失分。
   （這是「能推導就不存」原則的**刻意例外**——跨層共用時「維持清楚不變量」勝過「省一個推導
   欄位」；且復原是 rally+event 一起刪，winner 與 outcome 不會各走各的，無不同步風險。）
8. **記錄體感 = 節奏遊戲。** 基礎版計分操作的設計目標是「像玩節奏遊戲」——快、有打擊感、
   即時回饋、視線不離球場、維持心流。這是定位「簡易版操作成本不得高於紙筆」的操作層具現化，
   也是把 quality/座標/serveType 全推到進階版的理由（它們需要停下來瞄準、會打斷節拍）。
   餵給 T2（#74）：成本預算不只算點擊數，還要衡量「能否維持節拍、視線能否不離場」。

### 仍待決定

9. **到位門檻（幾分算到位）**：暫定 `quality >= 2`，與第 10 點「嗆司定義」一體兩面，兩者都
   **先用預設值、之後與教練確認再鎖**（原始 `quality` 有存，門檻永遠能重算，不急）。
10. **「嗆司」精確定義**（對方失誤送來的球？還是對方主動放的輕鬆球？）：影響一球怎麼歸到
    `ballType=chance`（D4 分母）。同上，先預設、之後定。
