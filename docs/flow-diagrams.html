輪轉表 — 已知問題與缺口

R1
刪除球員不會清掉場上殘留站位編輯名單彈窗存檔時只呼叫 setRoster()，不會順便清掉 rotations[].positions 裡指向已刪除球員 id 的殘留紀錄。Court.tsx 因為找不到球員會直接不渲染，畫面上看起來正常，但那個格子其實還卡著一個「幽靈站位」——下次有人被拖進同一格時才會被默默蓋掉，中間這段時間那格既看不到也選不到。
RosterEditDialog.tsx onSubmit → useRotationTable.ts setRoster() · issue #35
R2
移除球員是兩個 store 各自的動作右鍵刪除/拖出下緣，會同時呼叫輪轉表的 removePlayerFromCourt 跟戰術板的 removePlayerTacticPositions。這兩個呼叫寫在同一個元件裡目前沒問題，但如果之後有第三個地方也要「移除球員」，很容易漏呼叫其中一個，留下殘影。
components/PlayerNode.tsx · removeFromCourt() · issue #36
R3
「重置站位」按鈕放在輪轉表面板，實際操作兩個 store目前只有「全部清空」跟「不清」兩個選項，沒有「只清站位、保留畫筆」這種中間選項。
components/RotationTable.tsx · handleResetRotation() · 確認彈窗已在 issue #29 修正
R4
沒有要求排滿 6 個人才算「排好」hasRotations 只檢查「至少一輪有站位」，這個寬鬆判斷也是計分表 hasLineup 的資料來源，兩邊共用同一個缺口。

Part 2

戰術板 — 操作流程與狀態機

自上次更新以來的變化（2026-07-03）：戰術板從「讀輪轉表即時站位當背景」改成「進入戰術布置那一刻拍一張快照，之後完全獨立」——下面內文提到「讀輪轉表站位當背景」的地方已經不準確，實際行為是：點「戰術布置」→ `enterTacticsLayout()` 複製一份站位 → 之後編輯不再寫回輪轉表。自由球員在戰術布置裡也不再檢查「只能站後排」，變成一般標記。連帶：入口按鈕改成「戰術布置」（用輪轉表重拍）/「編輯」（沿用已載入的已儲存戰術，需先點清單才會亮起）；布置模式底部是「取消／儲存／另存新檔」；畫筆物件（箭頭/文字/排球）原本不能拖曳移動的 bug 已修正。舊的「已知問題」清單裡 T3、T5 已經因為這次改動而不再適用，見下方標註。

戰術板只有兩種模式：唯讀檢視、跟戰術布置。大部分操作只在布置模式裡才能做，進入布置時會把輪轉表當下的站位拍一張快照，之後編輯完全獨立，不會寫回輪轉表。
