Part 2

戰術板 — 操作流程與狀態機

自上次更新以來的變化（2026-07-20）：戰術板經過 issue #154 與 #160 C2/C3 兩輪重構， 舊描述裡的 isLayoutMode、enterTacticsLayout()、以及「布置模式會寫回 /清掉輪轉表資料」等機制已經不存在（不只是過期，是被刪掉了）。現在的形狀是：戰術板持有 一個「用完即丟的單景 session」，session 裡的球員是反正規化的快照（姓名/背號/位置在擷取 當下就凍進去，沒有指回 roster 的外鍵），所以「不會寫回輪轉表」不再靠自律，而是型別上根本沒有 能寫回去的東西（capture by value）。這條單向性由 ESLint no-restricted-imports 焊進 CI——useTacticsBoard.ts 不准 import useRotationTable / useScoreSheet。

戰術板有三種模式，但沒有任何 mode / phase 欄位——模式是從 store 既有的兩個欄位推導出來的（見 TacticsBoardPanel.tsx）： session !== null → 編輯；viewingScene !== null（且無 session）→ 唯讀檢視；兩者皆 null → 瀏覽。刻意不加中間的「排列中」狀態：輸入一張快照、開始編輯，這件事 本身就是 startSession() 一次做完的。

自上次更新以來的變化（2026-07-20）：新增暫停（C2b，issue #44，S5 因此不再是缺口）＋右欄底部「快速戰術板」入口（C5，issue #160 C3）；頁面左側的「回列表／戰術板」連結換成三頁共用的 MatchNavRail 導覽軌（issue #160 C1），不再是這頁 header 自己刻的。 更早（2026-07-12）：C1 動作選單依目前發球方過濾「發球↔接發」互斥（發球方看不到「接發」、接發方看不到「發球」，其餘四動作保留，#50）。 更早（2026-07-04）：記一球的手勢從「拖曳畫線」改成「直接點選」（放開時一樣是找命中誤差半徑內最近的目標，點跟拖曳結果相同，只是不強迫使用者拖曳）；動作選單從 4 大類（發球/防守/攻擊/擋網）擴充成 6 大類（發球/接發/舉球/攻擊/攔網/防守），對齊 `events` 表的 `eventActionEnum`；點選目標新增「對手(全體)」（不挑對手是哪個球員/位置）；新增「沒看到」按鈕，跳過選動作直接記得/失分，不建立任何動作紀錄；得/失分改成相對於「這球的動作方」判定，不再永遠對應我方。下面的 C1 內容已經更新反映這些變化。

照畫面實際會經過的順序排列。C 是整場比賽大部分時間停留的地方，裡面四條路徑（得分／復原／換人／自由球員）彼此獨立，但都會回到同一個「Rally Idle」畫面。

C

Rally Idle — 一局進行中的主畫面

四條操作路徑都從這裡出發，也都回到這裡。球場可互動的前提是沒有選單開著、沒有正在拖自由球員。

C1 · 記一球（點選手勢）
點選我方球員 / 對手(全體)
→
動作選單：發球/接發/舉球/攻擊/攔網/防守（6 大類，依發球方過濾發球↔接發 #50）
→
結果選單：得分/失分（相對這球的動作方判定）
→
scorePoint() — 記分、可能換輪轉、換發球方
或：點「沒看到」按鈕
→
直接跳結果選單（跳過選動作）
→
scorePoint() — 只記分，不建立任何動作紀錄
C2 · 復原（逐動作）
「復原」按鈕（復原堆疊為空時鎖住）
→
undoLast() — 一次退最近一個動作（得分／一般換人／手動 libero／暫停），連按往回
C2b · 暫停（issue #44）
點「我方暫停 n/2」或「對手暫停 n/2」
→
記一筆 timeout（setId ＋ 當下比分快照 ＋ side），本地優先＋背景寫入
→
達 2 次該側按鈕反灰（硬規則，無例外）
C3 · 一般換人
點板凳球員
→

→
