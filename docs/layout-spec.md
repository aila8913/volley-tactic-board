# 版面配置規格（Layout Spec）

來源：Figma `Volley Tactics Board`
（<https://www.figma.com/design/BdGsRuFoHYhiQVQABt9A5g/Volley-Tactics-Board>），
2026-07-21 由 Figma 檔案結構＋截圖整理。

**這份文件管「東西擺在哪、多寬、怎麼開合」**；顏色、字體、玻璃質感、發光那些
「長什麼樣」的規則在 [design-spec.md](./design-spec.md)。Figma 這份是**灰階線框稿**
（wireframe），刻意只有灰白黑——所以它對版面有話語權，對配色沒有。兩者衝突時：
版面聽這份、視覺聽 design-spec。

> 線框稿裡的文字多半是縮寫佔位（「比 / 計 / 數 / 戰」「資夾」「篩」），代表的是
> 「這裡有一個入口」，不是最終文案。

## 0. 畫布基準與換算

所有 frame 都是 **2048 × 1440**（桌機橫向，約 16:11）。下面的 px 都是這個基準下的值。
實作時**不要硬寫 px**，改用比例或 Tailwind 尺標——桌機解析度千百種，寫死 2048 下的
數字換到 1440 寬的筆電就會爆版。轉換方式：

| 用途                     | Figma px | 佔畫布寬 | 建議實作                                |
| ------------------------ | -------- | -------- | --------------------------------------- |
| 外框邊距                 | 28       | 1.4%     | `p-6` / `p-7`（固定值即可，不用等比縮） |
| 欄與欄之間的溝（gutter） | 40–43    | 2%       | `gap-8` / `gap-10`（同上，固定值）      |
| 導覽軌（收合）           | 105      | 5.1%     | 固定 `w-24`～`w-28`                     |
| 導覽側欄（展開）         | 370      | 18.1%    | 固定 `w-[22rem]`                        |
| 右側資訊欄               | 493      | 24.1%    | 固定 `w-[30rem]`，窄螢幕再收合          |
| 右側工具軌（收合）       | 132      | 6.4%     | 固定 `w-32`                             |
| 中央主區                 | 剩餘     | 64%±     | `flex-1`，吃掉剩下的空間                |

**原則：左右兩側用固定寬，中央 `flex-1` 吃剩餘**。因為側欄裝的是清單／按鈕（內容寬度
需求穩定），中央裝的是球場／列表（越寬越好用）。反過來讓側欄等比縮，小螢幕上按鈕會
先擠爆。

垂直方向：內容區從 `y≈110–124` 開始、高 `1321`，上下各留約 110px。等同「頂部留白
≈ 底部留白，主區佔滿中間」，實作用 `h-screen` + `py-*` 即可，不用抄絕對值。

## 1. 三欄骨架（全站共用）

每一張稿都是同一個骨架，只差在「某一欄是展開還是收合」：

```
┌──────┬─────────────────────────────┬──────────────┐
│ 導覽 │        中央主區              │  右側資訊欄  │
│ 軌   │  （列表 / 球場，flex-1）     │ （輪轉表＋   │
│ 105  │                              │   球員清單） │
│ 或   │                              │   493        │
│ 370  │                              │   或 132     │
└──────┴─────────────────────────────┴──────────────┘
   28      gutter 40           gutter 40        28
```

三欄各自的收合狀態組合出四種版面模式：

| 模式       | 左欄     | 中央          | 右欄           | 對應 Figma frame  |
| ---------- | -------- | ------------- | -------------- | ----------------- |
| A 列表瀏覽 | 收合 105 | 列表 1310     | 資訊欄 493     | Frame 1 / Frame 6 |
| B 戰術唯讀 | 展開 370 | 球場 1047     | 資訊欄 493     | Frame 2           |
| C 戰術編輯 | 展開 370 | 球場 **1406** | 工具軌 **132** | Frame 3           |
| D 對手佈陣 | 展開 370 | 球場 1047     | 資訊欄 493     | Frame 5           |

**模式 C 是重點**：一進入編輯，右側資訊欄（輪轉表／球員清單）整條換成 132px 的工具軌，
中央球場區從 1047 撐到 1406（+34%）。這是刻意的——編輯時人只看球場，輪轉表和名冊
是「決定要編什麼」階段才需要的東西，留著只會偷走畫布。**唯讀↔編輯不是加一排工具列，
是整個右欄換人**。

## 2. 左欄：導覽軌 / 導覽側欄

### 2.1 收合態（105px）

- 直排單字入口，由上而下：**比**（比賽列表）、**計**（計分表）、**數**（數據分析）。
- **戰**（戰術板）不跟前三個排在一起，**單獨貼在最底部**（Frame 1 裡 y=1336，
  離其他三個超過 900px）。這個位置差異是有意義的：前三個是「比賽資料」的階層，
  戰術板是另一個維度的工具入口。
- 滑鼠 hover 時展開成側欄（Figma 紅字註記：「hover 拉出側邊欄位」）。

### 2.2 展開態（370px）

- 入口變成完整文字：「比賽列表 / 計分表 / 數據分析」，字級明顯放大（58px 高的字框）。
- **當前所在的入口整條反白**（黑底白字，見 Frame 2 的「戰術版」、Frame 3 的「戰」）——
  active 狀態是整行 highlight，不是只換文字顏色。
- active 的入口**底下直接展開它的子清單**：戰術板底下列出「基本輪轉 1…」七筆，左側有
  一條垂直細線串起來（`Vector 3`，高 499px），視覺上表達「這些屬於上面那一項」。
- 子清單項目高 58px、寬 277px，選中項用淺灰底標示（比 active 入口的黑底弱一階——
  兩層選取狀態要有強弱差，不然分不出「在哪一頁」和「看哪一筆」）。
- 清單最底下是 **`+` 新增列**（Frame 4/5），寬高與一般項目相同。

> 註記提示：「隨時拉出戰術版做使用」「檢視現有戰術開啟唯讀模式」——從側欄點任一筆
> 戰術，中央直接進**唯讀**模式，不是編輯模式。

## 3. 中央主區

### 3.1 列表型（模式 A）

- 容器 1310 寬，內縮後內容寬 1136。
- 右上角一排操作按鈕，由左至右：**篩**（87×89，方形）、**資夾**（148×89）、
  **比賽**（148×89）。方形的是圖示鈕，長方形的是帶文字的建立鈕。
- 列表卡片 **1136 × 252**，**卡距 53px**（629→934→1239 每隔 305，扣掉高度 252）。
  卡片很高、間距很鬆——這是給「賽場邊快速點選」用的尺寸，不是資訊密度優先的表格。
- 卡片內一行式排版：
  `[類型徽章 87×89] [主標題] ………… [日期] [次要資訊]`
  - 徽章：**資**＝賽事資料夾、**比**＝單場比賽。同一個列表混合兩種項目，靠徽章區分。
  - 主標題左緣 x=458（距卡片左緣 200px），日期／次要資訊靠右對齊。
  - 次要資訊隨類型變：資料夾顯示「5 場比賽」，比賽顯示「3:0 勝」。
- 列表右側有一條 **8px 寬的捲軸指示條**（`Rectangle 27`），長度 478——是自訂捲軸，
  不是瀏覽器預設。
- **選中的卡片整張反白**（Frame 6 的黑底白字），與側欄 active 同一套語言。

### 3.2 球場型（模式 B/C/D）

球場本身固定 **457 × 914**，比例正好 **1:2**——對應真實排球場 9m × 18m，
所以球場尺寸不能隨手改成別的比例。內部三條橫線：

| 線               | 距球場頂  | 佔全長  | 意義           |
| ---------------- | --------- | ------- | -------------- |
| `Vector 6` y=609 | 279 / 914 | 30.5%   | 上方 3m 攻擊線 |
| `Vector 4` y=769 | 439 / 914 | 48%≈50% | 中線（網）     |
| `Vector 5` y=926 | 596 / 914 | 65%     | 下方 3m 攻擊線 |

兩條攻擊線對中線上下對稱（±160px ≈ 全長 17.5%）。實作務必用**比例**畫，
用固定 px 會在不同容器寬度下跑掉。

球場周邊固定元素：

- **左上 96×96 方塊**、**右下 96×96 方塊**——兩個對角的場外區塊（發球位／替補區之類），
  位置固定不隨球場縮放漂移。
- 右上角**模式按鈕 149×69**：唯讀時顯示「編輯」（淺底），編輯中顯示「編輯中」
  （**黑底反白**）。註記寫明「點選編輯才可編輯」——**預設唯讀，編輯是明確切換**。
- 佈陣時球員為 **直徑 71 的圓**，排成 3 欄 × 2～3 列（欄 x=762/919/1076，
  列 y=427/620/831）。我方與對方用不同色（Frame 5：綠 vs 粉），對應右側球員清單
  同樣染色。
- 球（`Ellipse 1`，直徑 64）落在球場中線附近，是可拖曳物件。

## 4. 右欄：資訊欄（493px）

由上而下三塊，順序固定：

1. **輪轉表**（標題右對齊）
   - 表格 **377 × 247**，3 欄 × 2 列。
   - 位置編號排法：上排 `4 3 2`、下排 `5 6 1`——**這是從場上視角看的站位**，
     1 號在右後（發球位），逆時針 1→6→5→4→3→2。編號位置不可自行調整順序。
   - 表格內的格子可以直接放球員（Frame 6 左下格變成一顆黑圓點）。
2. **輪次切換列**（375 × 69）
   - `[上] 115×69` … `第一輪`（置中文字） … `[下] 115×69`
   - 上／下兩鈕貼齊左右端，當前輪次文字在中間——是 stepper 不是下拉選單。
3. **球員清單**（標題「球員」右對齊）
   - 列高 **55**、寬 376，列距 12（753.5 → 820.5 → 891.5）。
   - 最後一列是 **`+` 新增列**，樣式與一般列相同、內容只有置中的 `+`。
   - 佈陣模式下這裡的列會依隊伍染色（Frame 5 粉色＝對手）。

**跨欄互動**：Frame 6 的紅色箭頭標示「拖曳」——從球員清單的某一列拖到輪轉表的格子裡
完成上場。所以球員清單列與輪轉表格子都要是拖放目標，且兩者要在同一個 DnD context。

## 5. 右欄：編輯工具軌（132px，模式 C）

取代資訊欄的窄軌，兩段式：

- **上段（y 156–1028，約 872 高）**：繪圖工具列——箭頭／格線／矩形／曲線／連線／
  圓角矩形／群組，下方以分隔線隔開另一組（畫筆／文字／圖層／排序）。每個工具下方帶一個
  小三角，代表**可展開次選項**。目前 Figma 裡這段是貼圖佔位（`image 2`），
  圖示語彙尚未定案。
- **下段（y 1100 起）**：三顆 80×80 方鈕，垂直排列、間距 23：
  **存**（儲存）／**另**（另存新檔）／**取**（取消）。
  貼在軌道底部，與繪圖工具明顯分離——避免「畫一畫誤觸取消」。

## 6. 新增戰術對話框（Frame 4）

從側欄 `+` 進入時出現：

- 遮罩覆蓋中央球場區，**不覆蓋左右欄**（`Rectangle 17` 791 × 512，蓋在球場上）。
  這是刻意的——建立戰術時仍看得到自己在哪一頁、右邊有哪些球員。
- 對話框內兩顆並排選項鈕（各 185 × 112，間距 87）：
  - **現有輪轉位**——沿用目前輪轉表的站位
  - **重新佈陣**——註記：「重新佈陣的話直接拉球員上場」，即空場，從右欄拖人進來
- 選完後進入 Frame 5 的佈陣流程，右上角按鈕變成「**確定**」（149×69，與「編輯」同尺寸
  同位置——是同一顆按鈕換文案，不是新增一顆）。

## 8. 落地計畫：M1.5 的七個環

### 為什麼要分環

三欄骨架是**跨頁面的共用約束，但程式裡沒有主人**。`App.tsx` 只有路由沒有 shell，
`ScoreSheet.tsx` 與 `MatchAnalytics.tsx` 的註解都寫著「跟另一頁一樣改成 `flex h-screen`」
——靠複製貼上維持一致性。結果是**每開一個 UI issue 都要重新決定一次「這頁三欄長怎樣」**，
所以 UI 工作一直是碎的。

解法不是把它們合成一個大任務（那只是換一種沒系統），而是**先讓版面有主人**，
之後每一環都只是「往某個插槽填內容」，彼此不會互相踩。

依賴鏈：`環0 → 環1 → (環2 ‖ 環3 ‖ 環4) → 環5 → 環6`。
環 2／3／4 各管左／右／中，互不相干，順序可調。

| 環  | 內容                                                        | 對應段落 | Issue                                                                                    |
| --- | ----------------------------------------------------------- | -------- | ---------------------------------------------------------------------------------------- |
| 0   | 視覺語言定案（選取態在深色下長什麼樣）                      | §7       | 併入 [#134](https://github.com/aila8913/volley-tactic-board/issues/134)                  |
| 1   | **版面骨架**：抽出 `AppShell`，三欄與四種模式收斂到單一元件 | §0 §1    | [#172](https://github.com/aila8913/volley-tactic-board/issues/172)                       |
| 2   | 左欄導覽：收合 ↔ hover 展開、active 整行反白、戰術子清單    | §2       | [#173](https://github.com/aila8913/volley-tactic-board/issues/173)                       |
| 3   | 右欄資訊欄：輪轉表元件化 ＋ 球員清單 ＋ 跨欄拖曳            | §4       | [#174](https://github.com/aila8913/volley-tactic-board/issues/174)                       |
| 4   | 中央列表型：大卡片、資／比 徽章、選中整張反白               | §3.1     | [#175](https://github.com/aila8913/volley-tactic-board/issues/175)                       |

### 環 1 的範圍限制（重要）

環 1 **只定契約，不重寫功能元件**。理由：若它變成「盤點程式每個 part 做耦合／解耦」，
會膨脹成做不完的大工程——等於用一個超大任務取代一堆碎任務，一樣沒系統。
而且現在猜「輪轉表需要哪些 prop」一定會猜錯，只有做到環 3 才會知道。

**功能元件的解耦在各自的環裡做**：輪轉表＝環 3、球場＝環 5。
環 1 蓋隔間和插座，環 3／5 才決定插座上接什麼電器。

環 1 也**不做任何視覺變更**——搬家和裝潢混在一起，出事時分不出是搬壞了還是裝潢壞了。
**改版前後畫面沒有明顯差異，是這一環成功的標誌，不是缺點。**

### 尚未由這份線框稿回答的問題

- **手機／平板版面沒有稿**。三欄骨架在窄螢幕怎麼塌（右欄變 bottom sheet？導覽變底部
  tab？）尚無定案。產品定位是 PWA 優先，這塊需要補——排在環 7（#178，M3）。
- **繪圖工具軌的圖示語彙**是貼圖佔位，尚未定案。環 5（#176）動手前需要 `@tangyi1025` 補圖示。
- **文案**全部是佔位縮寫，正式文案未定。
- 線框稿是灰階，**與 design-spec 的深色＋萊姆綠配色如何對應**（「整條／整張反白」的
  選取態在深色主題下是萊姆綠實底、還是玻璃提亮＋細環）需要再定一次——
  這就是環 0，已併入 #134 的視覺定案討論。注意選取態有**兩層強弱**
  （導覽 active 強 ／ 子清單選中弱一階，見 §2.2），兩層都要定。
